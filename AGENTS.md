# AGENTS.md

이 문서는 **presen2U**(단일 HTML 파일) 프로젝트에서 작업하는 AI 에이전트와 개발자를 위한 지침입니다.
모든 코드, 주석, 커밋 메시지, 응답은 **한국어**를 기본으로 합니다.

---

## 1. 프로젝트 개요

- **형태**: 단일 HTML 파일 (`index.html`) 하나로 완결되는 프로젝트
- **원칙**: 별도의 빌드 도구, 번들러, 외부 프레임워크 없이 브라우저에서 파일을 열면 바로 동작해야 한다.
- **의존성**: 외부 라이브러리는 되도록 사용하지 않는다. 꼭 필요하면 CDN `<script>`/`<link>` 한 줄로 불러오고, 그 이유를 주석으로 남긴다.

---

## 2. 파일 구조

### 프로젝트 디렉터리 규칙

- **`index.html`은 프로젝트 루트에 두는 메인 파일**이다. 진입점 역할을 한다.
- **메인을 제외한 모든 기능은 각각 별도 폴더로 관리**한다. 기능 하나 = 폴더 하나.
- 폴더 이름은 기능을 알 수 있게 소문자-하이픈으로 짓는다. 예: `gift-list/`, `budget/`

```
presen2U/
├── index.html          ← 메인 진입점 (루트 고정)
├── AGENTS.md
├── gift-list/          ← 기능: 선물 목록
│   ├── index.html      또는 gift-list.html
│   ├── style.css
│   └── script.js
├── budget/             ← 기능: 예산 관리
│   └── ...
└── (기능마다 폴더 추가)
```

### 메인 HTML 내부 구조

메인 `index.html` 안에서도 역할을 명확히 구분한다. 순서를 지켜 가독성을 유지한다.

```
index.html
├── <head>
│   ├── <meta> 설정 (charset, viewport 등)
│   ├── <title> 문서 제목
│   └── <style> ─ 모든 CSS (섹션 주석으로 구획)
└── <body>
    ├── 마크업 ─ 시맨틱 태그(header, main, section, footer) 사용
    └── <script> ─ 모든 JS (기능 단위 함수로 분리)
```

### CSS 구획 규칙
`<style>` 내부는 아래 순서로 주석 구획을 나눈다.

```css
/* ===== 1. 전역 변수 (CSS 변수) ===== */
/* ===== 2. 기본/리셋 스타일 ===== */
/* ===== 3. 레이아웃 ===== */
/* ===== 4. 컴포넌트 ===== */
/* ===== 5. 반응형 (미디어 쿼리) ===== */
```

### JS 구획 규칙
`<script>` 내부는 아래 순서로 구성한다.

```js
/* ===== 1. 상수/설정 ===== */
/* ===== 2. 상태(state) ===== */
/* ===== 3. 순수 함수 (계산/변환) ===== */
/* ===== 4. DOM 조작 함수 ===== */
/* ===== 5. 이벤트 핸들러 ===== */
/* ===== 6. 초기화(init) ===== */
```

## 3. 한국어 주석 규칙

- **모든 주석은 한국어로 작성**한다. 영어 기술 용어는 괄호로 병기할 수 있다. 예: `// 상태(state)를 초기화한다`
- **"무엇을"이 아니라 "왜"를 설명**한다. 코드만 봐도 아는 내용은 주석으로 반복하지 않는다.
  - 나쁨: `// i를 1 증가시킨다`
  - 좋음: `// 사용자가 마지막 슬라이드를 넘기면 처음으로 되돌린다`
- **구획 주석**은 `/* ===== 제목 ===== */` 형식으로 통일한다.
- **함수 주석**은 함수 바로 위에 한 줄로 역할을 적는다. 복잡한 함수는 매개변수/반환값을 덧붙인다.

```js
// 선물 목록을 받아 총 예산 금액을 계산한다
// gifts: {name, price}[] → number
function calcTotalBudget(gifts) {
  return gifts.reduce((sum, g) => sum + g.price, 0);
}
```

- **TODO / FIXME**도 한국어로 남긴다. 예: `// TODO: 빈 목록일 때 안내 문구 표시`

## 4. 함수 분리 원칙

- **하나의 함수 = 하나의 역할** (단일 책임). 함수 이름만 읽어도 무슨 일을 하는지 알 수 있어야 한다.
- **계산과 표시를 분리**한다.
  - **순수 함수**: 입력만으로 결과가 정해지고 DOM/전역 상태를 건드리지 않는 함수 (계산·변환·검증).
  - **DOM 함수**: 화면을 그리거나 바꾸는 함수 (`render*`, `update*`, `show*`).
  - **핸들러**: 이벤트를 받아 위 함수들을 조립하는 함수 (`onClick*`, `handle*`).
- **함수 길이**는 화면 한 눈(약 20~30줄)에 들어오도록 유지한다. 길어지면 쪼갠다.
- **명명 규칙**
  - 동사로 시작: `calc-`, `render-`, `update-`, `get-`, `is-/has-`(불리언 반환)
  - camelCase 사용. 예: `renderGiftList`, `isBudgetOver`
- **전역 변수 최소화**: 상태는 한 곳(`state` 객체 등)에 모으고, 함수는 인자로 값을 주고받는다.
- **중복 제거**: 같은 로직이 두 번 나오면 함수로 뽑는다.

## 5. Git 플로우

모든 기능 추가·수정은 **브랜치를 분리해서 구현하고, 완료 후 `main`에 병합(merge)**하는 방식으로 진행한다.
`main`에 직접 커밋하지 않는다.

### 5.1 전체 흐름 한눈에

```
main 최신화 → 기능 브랜치 생성 → 구현(변경마다 commit) → main pull 후 merge(로컬에서 충돌 해결) → GitHub push
```

### 5.2 브랜치 규칙

- 브랜치 이름: `<타입>/<기능-이름>` (소문자-하이픈). 타입은 커밋 타입과 동일.
  - 예: `feat/gift-list`, `fix/budget-overflow`, `refactor/render-split`
- 기능 하나당 브랜치 하나. 서로 다른 기능을 한 브랜치에 섞지 않는다.

### 5.3 기능 시작 (브랜치 분리)

작업을 시작하기 전, `main`을 최신 상태로 맞춘 뒤 브랜치를 만든다.

```bash
git checkout main
git pull origin main          # main을 최신으로 동기화
git checkout -b feat/기능-이름  # 기능 브랜치 생성
```

### 5.4 구현 중 (변경마다 커밋)

- **변경사항이 생길 때마다 그때그때 `git commit`**을 진행한다. 한 번에 몰아서 커밋하지 않는다.
- 커밋 메시지는 아래 [6. 커밋 규칙](#6-커밋-규칙)을 따른다.

```bash
git add <변경한 파일>
git commit -m "feat: 선물 목록 렌더링 함수 추가"
# ... 작업 이어가며 논리적 단위마다 반복 ...
```

### 5.5 기능 완료 후 병합 (안전한 merge)

기능 구현이 끝나면 `main`을 먼저 당겨와 최신 상태에서 병합한다. **충돌은 로컬에서 해결**한다.

```bash
# 1) 기능 브랜치의 마지막 변경까지 커밋 완료 상태인지 확인
git status

# 2) main을 최신으로 갱신
git checkout main
git pull origin main

# 3) main에서 기능 브랜치를 병합 (충돌 발생 시 로컬에서 해결)
git merge feat/기능-이름
#   - 충돌이 나면 해당 파일을 열어 충돌 표시(<<<<<<< ======= >>>>>>>)를 정리하고
#     git add <파일> → git commit 으로 병합을 마무리한다.
#   - 충돌이 없으면 자동으로 병합 커밋이 생성된다.
```

> ⚠️ **충돌 해결은 반드시 로컬에서** 수행한다. 원격에 강제로 밀어붙이지(`--force`) 않는다.

### 5.6 병합 후 원격 반영 (push)

`main`으로 병합이 끝난 뒤에만 GitHub에 push 한다.

```bash
git push origin main
```

- push는 **merge가 완료되고 로컬에서 동작을 확인한 이후**에만 진행한다.
- 병합이 끝난 기능 브랜치는 정리할 수 있다: `git branch -d feat/기능-이름`

### 5.7 요약 표

| 단계 | 시점 | 명령 |
|------|------|------|
| main 최신화 | 기능 시작 전 | `git checkout main && git pull origin main` |
| 브랜치 분리 | 기능 시작 | `git checkout -b feat/기능-이름` |
| 커밋 | 변경사항이 생길 때마다 | `git add … && git commit -m "…"` |
| 병합 | 기능 완료 후 | `git checkout main && git pull && git merge feat/기능-이름` (충돌은 로컬 해결) |
| push | 병합 완료 후 | `git push origin main` |

## 6. 커밋 규칙

한 커밋 = 하나의 논리적 변경. 메시지는 **한국어**로 작성한다.

### 형식

```
<타입>: <한 줄 요약 (50자 이내, 마침표 없음)>

<본문 - 필요할 때만, 왜 바꿨는지 설명>
```

### 타입

| 타입 | 용도 |
|------|------|
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `style` | 화면/CSS 등 시각적 변경 (기능 영향 없음) |
| `refactor` | 동작은 그대로, 코드 구조 개선 |
| `docs` | 문서 수정 (README, AGENTS.md 등) |
| `chore` | 설정, 잡일, 기타 |

### 예시

```
feat: 선물 목록에 예산 초과 경고 표시 추가

총 예산이 설정값을 넘으면 합계 금액을 빨간색으로 표시한다.
사용자가 지출을 한눈에 인지하도록 하기 위함.
```

```
fix: 마지막 슬라이드에서 다음 버튼이 멈추던 문제 수정
```

### 규칙 요약
- 요약 줄은 명령형/현재형으로, 무엇을 했는지 명확하게.
- 여러 성격의 변경을 한 커밋에 섞지 않는다.
- 커밋 전 브라우저에서 `index.html`을 열어 동작을 확인한다.

## 7. 작업 체크리스트

작업을 마치기 전 아래를 확인한다.

- [ ] 기능 작업을 별도 브랜치에서 진행했는가
- [ ] 변경사항마다 커밋했는가
- [ ] 파일을 브라우저로 열었을 때 오류(콘솔 에러) 없이 동작하는가
- [ ] 새/수정 함수가 단일 책임을 지키는가
- [ ] 주석이 한국어이며 "왜"를 설명하는가
- [ ] 구획 순서(CSS/JS)가 지켜졌는가
- [ ] 커밋 메시지가 한국어 규칙을 따르는가
- [ ] `main` 최신화 → merge(로컬 충돌 해결) → push 순서를 지켰는가
